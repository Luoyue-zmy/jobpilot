# Jobpilot 产品需求文档（PRD）

> 产品暂定名：**Jobpilot**  
> 中文定位：**AI 求职进程管理平台**  
> 文档版本：v1.0  
> 适用阶段：Web MVP → 移动端/App 扩展  
> 核心目标：解决应届生秋招过程中“投递不漏、进度不乱、面试可复盘、复习有依据”的问题。

---

## 1. 文档说明

### 1.1 文档目的

本文档用于定义 Jobpilot 第一阶段 Web 端 MVP 的产品目标、功能范围、业务流程、数据模型、页面结构、接口需求、非功能需求和后续迭代方向。该文档可直接作为项目仓库中的 `docs/PRD.md`，用于指导后续基于 Codex / AI Coding 的模块化开发。

### 1.2 产品阶段

Jobpilot 采用分阶段开发策略：

- **第一阶段：Web 端 MVP**  
  完成岗位投递管理、JD 解析、进度看板、面试复盘、个人题库和轻量 AI 辅助，形成完整业务闭环。

- **第二阶段：移动端适配 / PWA**  
  对 Web 页面进行响应式适配，使用户可以在手机浏览器查看今日待办、快速修改状态、记录面试问题。

- **第三阶段：App 端扩展**  
  基于统一后端 API 开发轻量 App，聚焦移动场景：今日待办、面试前 Briefing、面试后速记、状态快速流转。

### 1.3 产品边界

Jobpilot 不是招聘平台，不提供岗位聚合、自动海投、自动爬取全网岗位、社区经验分享或复杂简历生成器。Jobpilot 的定位是：

> 用户已经找到岗位后，Jobpilot 帮助用户管理投递流程、沉淀面试经验，并将岗位与面试数据转化为复习依据。

## 2. 产品背景

### 2.1 用户痛点

应届生在秋招/春招过程中通常会同时投递大量岗位，岗位来源分散，流程节点复杂，记录成本高。典型问题包括：

1. **投递记录容易遗漏**  
   用户在官网、国聘、智联、BOSS、公众号、学校就业网等多个渠道投递岗位，投递后经常忘记记录。

2. **重复投递难以发现**  
   同一家公司或同一岗位可能在多个平台重复出现，岗位名称和公司名称表达不统一，用户容易重复保存或重复投递。

3. **投递进度混乱**  
   不同公司处于待投递、已投递、测评、笔试、一面、二面、终面、Offer、拒绝等不同状态，单纯表格难以直观看清当前优先事项。

4. **面试后复盘不系统**  
   面试中被问到的 Java 基础、Spring、MySQL、Redis、项目、算法、HR 问题常常没有及时沉淀，导致相同问题反复答不好。

5. **投递和复习脱节**  
   用户投了很多岗位，但不知道 JD 中哪些技术要求出现频率最高，也不知道近期面试暴露了哪些短板。

6. **现有工具不足**  
   飞书多维表格、Excel、Notion 可以记录信息，但需要手动维护，缺少状态流转、面试复盘、AI 总结、个人题库和面试前准备能力。

### 2.2 产品机会

Jobpilot 的机会不在于做“大而全求职平台”，而在于做一个高频、轻量、可落地的个人效率工具：

- 用结构化数据管理岗位和投递进度；
- 用 AI 降低 JD 录入和面试复盘成本；
- 用看板和待办帮助用户每天明确下一步行动；
- 用题库沉淀机制把面试经历转化为复习资产；
- 用 Web 端先完成完整闭环，再扩展 App 端移动场景。

## 3. 产品定位

### 3.1 一句话定位

**Jobpilot 是一款面向秋招应届生的 AI 求职进程管理工具，围绕岗位投递、进度跟踪、面试复盘和个人题库沉淀构建完整闭环，帮助求职者减少重复记录与进度混乱，把面试经历转化为可复习、可迭代的个人知识资产。**

### 3.2 简短定位

**AI 驱动的投递进程与面试复盘管理平台。**

### 3.3 产品核心价值

- 投递不漏：统一管理不同平台、不同公司的岗位记录。
- 进度不乱：通过状态机和看板管理投递进程。
- 面试可复盘：结构化记录面试问题、回答、表现和改进点。
- 复习有依据：基于 JD 和历史面试问题生成复习清单。
- 简历有亮点：体现 Java 全栈开发、AI 工程化、数据建模、状态机设计和产品闭环思维。

## 4. 目标用户

### 4.1 核心用户

第一版只服务一类核心用户：

> 正在秋招/春招、同时投递多个平台和公司、需要管理投递进度与面试复盘的应届生。

### 4.2 典型用户画像

#### 用户画像 A：Java 开发岗应届生

- 身份：硕士/本科应届毕业生；
- 投递方向：央国企 Java 开发岗、银行科技岗、运营商、研究所、部分私企开发岗；
- 岗位来源：公司官网、国聘、智联、BOSS、公众号、学校就业网；
- 痛点：岗位多、流程长、记录容易漏、面试题复盘不系统；
- 使用目标：管理投递进度，沉淀面试问题，提高复习效率。

#### 用户画像 B：多方向投递的开发岗求职者

- 投递方向：Java 后端、全栈、测试开发、数据开发、AI 应用开发；
- 痛点：不同岗位 JD 差异较大，不清楚每类岗位最需要补哪些技能；
- 使用目标：通过 JD 解析和面试复盘，找到自己的技能短板。

## 5. 产品目标与成功指标

### 5.1 MVP 产品目标

第一阶段 Web MVP 需要完成以下目标：

1. 用户可以完整管理岗位投递记录；
2. 用户可以通过看板查看投递进度；
3. 用户可以粘贴 JD 并生成结构化岗位信息；
4. 用户可以记录每一轮面试及问题回答；
5. 系统可以将面试问题沉淀为个人题库；
6. AI 可以生成 JD 分析、面试复盘建议和面试前 Briefing；
7. 系统可以形成“岗位 → 投递 → 面试 → 复盘 → 题库 → 再复习”的闭环。

### 5.2 可量化指标

MVP 阶段可设置以下项目验收指标：

- 岗位记录新增耗时：从手动表格 2–3 分钟降低至 30 秒以内；
- 至少支持 10 种投递状态；
- 支持按照公司类型、岗位方向、状态、城市进行筛选；
- 支持每个岗位绑定多场面试；
- 支持每场面试记录多个问题与回答；
- 支持从 JD 中抽取不少于 5 类结构化字段；
- 支持生成岗位匹配分析和复习建议；
- 支持将面试问题自动汇总到个人题库；
- Web 端可通过 Docker Compose 本地一键启动。

## 7. 用户核心流程

### 7.1 岗位录入流程

1. 用户在招聘网站、公众号或学校就业网看到岗位；
2. 用户复制 JD 文本；
3. 用户进入 Jobpilot 的“新增岗位”页面；
4. 用户粘贴 JD；
5. 系统调用 AI 解析 JD；
6. 系统自动填充公司、岗位、城市、技能要求、岗位职责、学历要求、岗位方向；
7. 用户检查并补充投递链接、截止时间、来源平台；
8. 用户保存岗位；
9. 系统检查是否存在疑似重复岗位；
10. 若存在重复岗位，提示用户确认；
11. 保存成功后岗位进入“待投递”状态。

### 7.2 投递状态流转流程

1. 用户打开岗位详情；
2. 用户完成投递后，将状态从“待投递”改为“已投递”；
3. 系统自动生成状态事件记录；
4. 用户收到测评、笔试、面试通知后继续更新状态；
5. 每次状态变化都会记录时间线；
6. 看板页面自动按状态展示岗位；
7. 今日待办根据截止时间、面试时间和待复盘状态生成提醒。

### 7.3 面试复盘流程

1. 用户收到面试通知；
2. 用户在岗位详情下新增面试记录；
3. 用户填写面试轮次、时间、形式、面试官类型；
4. 面试前用户点击“生成面试前 Briefing”；
5. 系统基于 JD、岗位技能、历史题库生成复习建议；
6. 面试后用户记录问题、自己的回答和表现；
7. 用户点击“生成 AI 复盘”；
8. 系统输出改进回答、薄弱点、复习清单；
9. 面试问题自动沉淀到个人题库；
10. 用户后续可在题库中复习相似问题。

### 7.4 个人题库流程

1. 系统汇总所有面试记录中的问题；
2. 用户可按问题类型筛选；
3. 用户查看原始回答和 AI 优化回答；
4. 用户标记掌握状态；
5. 系统根据未掌握和高频问题生成复习清单；
6. 面试前 Briefing 优先引用未掌握问题。

## 8. 功能需求详述

## 8.1 用户模块

### 8.1.1 功能说明

MVP 阶段只支持基础登录和个人技能标签维护，不做复杂账号体系。

### 8.1.2 功能点

- 用户登录；
- JWT 鉴权；
- 获取当前用户信息；
- 编辑个人技能标签；
- 编辑求职方向；
- 编辑目标岗位类型。

### 8.1.3 用户字段

| 字段 | 说明 |
|---|---|
| id | 用户 ID |
| username | 用户名 |
| passwordHash | 加密密码 |
| displayName | 显示名称 |
| targetRoles | 目标岗位方向，如 Java 后端、AI 应用开发 |
| skillTags | 个人技能标签，如 Java、Spring Boot、MySQL、Redis |
| createdAt | 创建时间 |
| updatedAt | 更新时间 |

### 8.1.4 验收标准

- 用户可以登录系统；
- 登录后可以访问岗位、看板、面试模块；
- 未登录用户访问业务接口时返回未授权；
- 用户可以维护个人技能标签。

---

## 8.2 岗位投递管理模块

### 8.2.1 功能说明

岗位模块是 Jobpilot 的数据底座，负责管理用户保存的所有岗位与投递记录。

### 8.2.2 功能点

- 新增岗位；
- 编辑岗位；
- 删除岗位；
- 查看岗位详情；
- 搜索岗位；
- 按状态筛选；
- 按公司类型筛选；
- 按岗位方向筛选；
- 按城市筛选；
- 按来源平台筛选；
- 按截止时间排序；
- 修改投递状态；
- 查看状态流转时间线；
- 疑似重复岗位提示。

### 8.2.3 岗位字段

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| id | Long | 是 | 岗位 ID |
| userId | Long | 是 | 所属用户 |
| companyName | String | 是 | 公司名称 |
| jobTitle | String | 是 | 岗位名称 |
| city | String | 否 | 工作城市 |
| jobUrl | String | 否 | 投递链接 |
| sourcePlatform | String | 否 | 来源平台 |
| companyType | Enum | 否 | 公司类型 |
| jobDirection | Enum | 否 | 岗位方向 |
| jdText | Text | 否 | 原始 JD |
| extractedSkills | JSON | 否 | JD 中提取的技能 |
| deadline | DateTime | 否 | 截止时间 |
| status | Enum | 是 | 当前状态 |
| priority | Enum | 否 | 优先级 |
| notes | Text | 否 | 用户备注 |
| createdAt | DateTime | 是 | 创建时间 |
| updatedAt | DateTime | 是 | 更新时间 |

### 8.2.4 投递状态枚举

正常流程状态：

- `TO_APPLY`：待投递；
- `APPLIED`：已投递；
- `ASSESSMENT`：测评；
- `WRITTEN_TEST`：笔试；
- `FIRST_INTERVIEW`：一面；
- `SECOND_INTERVIEW`：二面；
- `FINAL_INTERVIEW`：终面；
- `HR_INTERVIEW`：HR 面；
- `OFFER`：Offer。

终止状态：

- `REJECTED`：拒绝；
- `GIVEN_UP`：放弃；
- `CLOSED`：已结束。

### 8.2.5 状态流转要求

每次修改岗位状态时，系统必须自动生成一条状态事件记录，记录内容包括：

- 岗位 ID；
- 原状态；
- 新状态；
- 操作时间；
- 操作人；
- 备注。

### 8.2.6 重复岗位检测规则

MVP 阶段采用规则检测，不引入复杂向量库。

当用户新增岗位时，系统检查已有岗位：

- 公司名称完全相同或高度相似；
- 岗位名称完全相同或高度相似；
- 城市相同或任一为空；
- 来源平台可以不同。

若命中疑似重复，系统不阻止保存，但提示用户：

> 该岗位可能已存在，请确认是否继续保存。

### 8.2.7 验收标准

- 用户可以新增、编辑、删除和查看岗位；
- 用户可以修改岗位状态；
- 每次状态变化都生成时间线事件；
- 用户可以按状态、公司类型、岗位方向、城市筛选岗位；
- 新增疑似重复岗位时系统给出提示。

## 8.3 JD 粘贴解析模块

### 8.3.1 功能说明

用户粘贴岗位 JD 后，系统调用 AI 对非结构化文本进行解析，自动生成结构化岗位信息。

### 8.3.2 输入

- 原始 JD 文本；
- 可选：投递链接；
- 可选：来源平台。

### 8.3.3 输出字段

| 字段 | 说明 |
|---|---|
| companyName | 公司名称 |
| jobTitle | 岗位名称 |
| city | 城市 |
| jobDirection | 岗位方向 |
| companyType | 公司类型 |
| responsibilities | 岗位职责 |
| requirements | 任职要求 |
| educationRequirement | 学历要求 |
| skillTags | 技能标签 |
| deadline | 截止时间 |
| matchHighlights | 与用户技能匹配点 |
| riskPoints | 风险点/短板 |
| reviewSuggestions | 推荐复习内容 |

### 8.3.4 AI 解析要求

AI 输出必须为结构化 JSON，后端需要对返回结果进行解析和容错处理。

若 AI 接口不可用，系统应提供 mock fallback，保证核心流程可运行。

### 8.3.5 验收标准

- 用户粘贴 JD 后可以点击“解析 JD”；
- 系统返回结构化字段；
- 用户可以一键将解析结果填入新增岗位表单；
- AI 不可用时系统返回 mock 示例数据，不影响演示；
- 解析结果允许用户手动修改。

## 8.4 进度看板模块

### 8.4.1 功能说明

看板模块是用户每天打开 Jobpilot 的首页，用于快速了解当前求职进度和当天待办事项。

### 8.4.2 页面组成

- 顶部统计卡片；
- 今日待办；
- Kanban 状态看板；
- 最近面试；
- 待复盘面试；
- 即将截止岗位。

### 8.4.3 统计卡片

至少展示：

- 总岗位数；
- 已投递数；
- 进行中数量；
- 面试中数量；
- Offer 数；
- 待复盘数量。

### 8.4.4 Kanban 列

建议按以下状态分列：

- 待投递；
- 已投递；
- 测评/笔试；
- 面试中；
- Offer；
- 已结束。

MVP 阶段可以不做拖拽，先通过按钮修改状态。

### 8.4.5 今日待办生成规则

今日待办来源包括：

- 今天截止的岗位；
- 未来 3 天内截止的岗位；
- 今天有笔试/面试的岗位；
- 面试已结束但未复盘的记录；
- 长时间未更新状态的岗位。

### 8.4.6 验收标准

- 用户进入首页可以看到关键统计数据；
- 用户可以查看不同状态下的岗位；
- 用户可以看到今日待办；
- 用户可以从看板跳转到岗位详情；
- 用户可以从待办跳转到相关岗位或面试记录。

## 8.5 面试记录与复盘模块

### 8.5.1 功能说明

每个岗位下可以绑定多场面试，每场面试下可以记录多个问题、回答和复盘内容。

### 8.5.2 面试记录字段

| 字段 | 说明 |
|---|---|
| id | 面试 ID |
| jobId | 所属岗位 ID |
| round | 面试轮次 |
| interviewTime | 面试时间 |
| interviewFormat | 电话/视频/线下 |
| interviewerType | HR/技术/主管/其他 |
| status | 未开始/已完成/已取消 |
| selfScore | 自我评分 |
| summary | 复盘总结 |
| aiFeedback | AI 复盘建议 |
| createdAt | 创建时间 |
| updatedAt | 更新时间 |

### 8.5.3 面试问题字段

| 字段 | 说明 |
|---|---|
| id | 问题 ID |
| interviewId | 所属面试 ID |
| questionText | 面试问题 |
| questionType | 问题类型 |
| myAnswer | 我的回答 |
| improvedAnswer | AI 优化回答 |
| weakness | 暴露短板 |
| reviewPriority | 复习优先级 |
| mastered | 是否已掌握 |

### 8.5.4 问题类型枚举

- Java 基础；
- JVM；
- Spring；
- MySQL；
- Redis；
- 计算机网络；
- 操作系统；
- 项目问题；
- 算法；
- HR 问题；
- 反问问题；
- 其他。

### 8.5.5 AI 复盘能力

用户点击“生成复盘建议”后，系统基于以下输入生成结果：

- 岗位 JD；
- 面试轮次；
- 面试问题；
- 用户回答；
- 用户自我评分；
- 个人技能标签。

输出包括：

- 本场面试整体表现总结；
- 答得较好的地方；
- 暴露的技术短板；
- 每个问题的改进回答；
- 下一次面试前复习清单。

### 8.5.6 验收标准

- 一个岗位可以添加多场面试；
- 一场面试可以添加多个问题；
- 用户可以填写自己的回答和复盘总结；
- AI 可以生成改进回答和薄弱点分析；
- 面试完成后可以标记为“已复盘”。

## 8.6 个人题库模块

### 8.6.1 功能说明

个人题库用于汇总所有面试中出现的问题，并沉淀为用户自己的复习资料。

### 8.6.2 功能点

- 自动汇总面试问题；
- 按问题类型筛选；
- 按公司筛选；
- 按岗位方向筛选；
- 按掌握状态筛选；
- 查看我的回答；
- 查看 AI 优化回答；
- 标记已掌握/需复习；
- 生成复习清单。

### 8.6.3 题库来源

题库问题主要来源于面试记录，也允许用户手动新增。

### 8.6.4 验收标准

- 用户可以在题库中看到所有历史面试问题；
- 用户可以按类型筛选问题；
- 用户可以标记掌握状态；
- 未掌握问题可以进入复习清单；
- 面试前 Briefing 可以引用题库内容。

## 8.7 面试前 Briefing 模块

### 8.7.1 功能说明

面试前 Briefing 用于在面试前快速生成一页准备材料，帮助用户在短时间内回顾重点。

### 8.7.2 输入

- 岗位信息；
- JD 技能标签；
- 公司类型；
- 岗位方向；
- 历史相似问题；
- 用户未掌握问题；
- 用户个人技能标签。

### 8.7.3 输出

- 公司与岗位摘要；
- JD 核心技能；
- 本岗位可能考察的问题；
- 历史面试中答得不好的相似问题；
- 面试前 30 分钟复习清单；
- 项目表达建议；
- HR 问题准备建议；
- 反问问题建议。

### 8.7.4 验收标准

- 用户可以在岗位详情页生成 Briefing；
- Briefing 内容与当前岗位相关；
- Briefing 能引用用户历史题库中的薄弱点；
- Briefing 结果可以保存并再次查看。

## 8.8 提醒与待办模块

### 8.8.1 功能说明

提醒模块用于生成求职过程中的关键待办，避免错过截止时间、笔试、面试和复盘。

### 8.8.2 待办类型

- 岗位即将截止；
- 今天需要投递；
- 今天有测评；
- 今天有笔试；
- 今天有面试；
- 面试后待复盘；
- 长时间未更新状态。

### 8.8.3 生成规则

- 截止时间在 3 天内且状态为待投递的岗位生成截止提醒；
- 面试时间为今天的记录生成面试提醒；
- 面试状态为已完成但未复盘生成待复盘提醒；
- 状态超过 14 天未更新的岗位生成跟进提醒。

### 8.8.4 验收标准

- 首页能展示今日待办；
- 用户可以标记待办完成；
- 用户可以从待办跳转到相关岗位或面试记录。

## 8.9 AI 服务模块

### 8.9.1 功能说明

AI 服务模块为 JD 解析、岗位匹配分析、面试复盘、面试前 Briefing 提供统一接口。

### 8.9.2 AI 能力列表

| 能力 | 输入 | 输出 |
|---|---|---|
| JD 解析 | JD 文本 | 结构化岗位字段 |
| 岗位匹配分析 | JD + 用户技能 | 匹配点、风险点、复习建议 |
| 面试复盘 | 问题 + 回答 + JD | 改进回答、薄弱点、复习清单 |
| 面试前 Briefing | 岗位 + 题库 + 技能 | 面试前准备摘要 |

### 8.9.3 AI 工程要求

- 使用统一 AI Client 封装大模型调用；
- 支持 OpenAI-compatible API；
- 支持通过配置切换 DeepSeek、通义千问或其他模型；
- 支持 mock 模式，避免没有 API key 时项目无法运行；
- AI 返回结果尽量要求 JSON 格式；
- 对 AI 返回结果做异常处理；
- 对相同输入的 AI 结果进行缓存。

### 8.9.4 缓存策略

使用 Redis 缓存以下结果：

- JD 解析结果；
- 岗位匹配分析结果；
- 面试复盘结果；
- 面试前 Briefing 结果。

缓存 key 可基于用户 ID、业务类型和内容 hash 生成。

### 8.9.5 验收标准

- AI 接口可正常调用；
- API key 不存在时 mock 模式可用；
- 重复请求相同内容时优先读取缓存；
- AI 返回异常时系统给出友好提示，不影响核心功能。

## 11. 接口需求建议

## 11.2 岗位接口

- `POST /api/jobs`：新增岗位；
- `GET /api/jobs`：岗位列表；
- `GET /api/jobs/{id}`：岗位详情；
- `PUT /api/jobs/{id}`：更新岗位；
- `DELETE /api/jobs/{id}`：删除岗位；
- `PUT /api/jobs/{id}/status`：修改岗位状态；
- `GET /api/jobs/{id}/events`：获取岗位状态时间线；

- `POST /api/ai/parse-jd`：解析 JD；
- `POST /api/ai/analyze-job-match`：岗位匹配分析。

## 11.4 面试接口

- `POST /api/jobs/{jobId}/interviews`：新增面试；
- `GET /api/jobs/{jobId}/interviews`：获取岗位面试列表；
- `GET /api/interviews/{id}`：面试详情；
- `PUT /api/interviews/{id}`：更新面试；
- `DELETE /api/interviews/{id}`：删除面试；
- `POST /api/interviews/{id}/questions`：新增面试问题；
- `PUT /api/interview-questions/{id}`：更新面试问题；
- `DELETE /api/interview-questions/{id}`：删除面试问题；
- `POST /api/interviews/{id}/ai-review`：生成 AI 面试复盘。

## 11.5 题库接口

- `GET /api/question-bank`：获取个人题库；
- `PUT /api/question-bank/{questionId}/mastered`：更新掌握状态；

## 11.6 Briefing 接口

- `POST /api/jobs/{jobId}/briefing`：生成面试前 Briefing；
- `GET /api/jobs/{jobId}/briefing/latest`：获取最近一次 Briefing。

## 11.7 看板与待办接口

- `GET /api/dashboard/summary`：获取统计卡片；

## 12. 非功能需求

### 12.1 性能需求

- 岗位列表查询响应时间应小于 500ms；
- 看板数据加载时间应小于 800ms；
- AI 接口允许较长响应时间，但前端需要展示 loading 状态；
- AI 结果应缓存，避免重复调用。

### 12.2 安全需求

- 用户密码必须加密存储；
- 使用 JWT 做接口鉴权；
- 用户只能访问自己的岗位和面试数据；
- AI 请求中避免上传不必要的敏感信息；
- 删除操作需要二次确认。

### 12.3 可用性需求

- 表单字段应提供默认值和下拉枚举；
- JD 解析结果应允许用户修改；
- AI 失败时不阻塞主流程；
- 页面应提供空状态提示；
- 移动浏览器下主要页面可基本使用。

### 12.4 可维护性需求

- 后端采用 controller/service/mapper/entity/dto 分层；
- 前端采用 pages/components/api/types 分层；
- AI 调用封装为独立 service；
- 状态枚举统一管理；
- 接口文档使用 Swagger / Knife4j。

- 一次只做一个模块；
- 不修改无关文件；
- 保持代码简单可读；
- 优先实现可运行闭环；
- 每个功能给出验证步骤；
- 后端遵循 controller/service/mapper/entity/dto 分层；
- 前端遵循 views/components/api/types 分层；
- AI 功能必须支持 mock fallback；
- 不引入第一版不需要的复杂架构。

推荐任务顺序：

1. 初始化后端；
2. 初始化数据库；
3. 实现岗位 CRUD；
4. 实现状态流转；
5. 初始化前端；
6. 实现岗位列表和详情；
7. 实现看板；
8. 实现 JD 解析；
9. 实现面试记录；
10. 实现 AI 复盘；
11. 实现题库；

## 18. 简历项目表述建议

项目名称：**Jobpilot｜AI 求职进程管理平台**

项目描述：

> 面向应届生秋招多平台投递记录分散、进度混乱和面试复盘不系统的问题，设计并实现 Jobpilot AI 求职进程管理平台。系统支持岗位投递管理、JD 智能解析、投递状态看板、面试记录复盘、个人题库沉淀和面试前 Briefing，帮助用户将求职过程从零散记录转化为可追踪、可复盘、可迭代的个人求职工作流。

技术栈：

> Spring Boot 3、MyBatis-Plus、MySQL、Redis、Spring Security、JWT、Vue 3、TypeScript、Element Plus、OpenAI-compatible API、Docker Compose。

项目亮点：

- 设计投递状态机，覆盖待投递、已投递、测评、笔试、多轮面试、Offer、拒绝等真实秋招流程，并通过事件表记录状态流转时间线；
- 基于大模型实现 JD 结构化解析，从非结构化岗位描述中抽取公司、岗位、城市、技能要求、岗位职责和复习建议，降低岗位录入成本；
- 构建面试复盘与个人题库机制，将面试问题、用户回答、AI 优化回答和薄弱点分析结构化沉淀为复习资料；
- 实现面试前 Briefing，根据岗位 JD、历史面试问题和个人技能标签生成面试前准备摘要；
- 使用 Redis 缓存 AI 分析结果，减少重复调用大模型接口，降低响应延迟和调用成本；
- 基于前后端分离架构和 Docker Compose 完成系统部署，形成可演示的 Web 端完整业务闭环。

## 19. 最终总结

Jobpilot 第一版的核心不是做招聘平台，而是做一个个人求职过程管理工具。产品应聚焦五个核心能力：

1. 岗位投递管理；
2. JD 粘贴解析；
3. 进度看板与今日待办；
4. 面试记录与 AI 复盘；
5. 个人题库与面试前 Briefing。

第一阶段先完成 Web 端完整闭环，第二阶段再将今日待办、状态快速修改、面试前 Briefing 和面试后速记同步到 App 端。该路线既符合真实使用场景，也能在简历中体现 Java 全栈开发、AI 工程化、状态机设计、数据建模和产品思维。
